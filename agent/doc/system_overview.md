# システム概要

JSAI 2026 向け人狼知能エージェント (`aiwolf-jsai-manyshot`) の仕様概要。発表・対外説明用の高位ビュー。
実装の詳細は [ARCHITECTURE.md](../ARCHITECTURE.md) や [doc/](.) 以下の各機能ドキュメントを参照。

---

## 1. 全体像: 3 段階のパイプライン

このシステムは **3 段階の準備 → 実ゲーム** という構成で、それぞれ異なる LLM が役割分担しています。

```
[1] 台本生成 (オフライン, 1 回限り)
       │  Claude Opus 4.7 が実プレイログを生成
       ▼
   data/sample_games_md/  に 5人村・9人村それぞれ 10 本の Markdown 台本

[2] 事前準備 (ゲーム前, モデル変更時に再実行)
       │  gpt-5.4 が台本を読み込んで分析・要約
       ▼
   ローカルキャッシュに「世界モデル」を保存

[3] 実ゲーム (本番)
       │  talk 系: gpt-5.2 / action 系: gpt-4o-mini
       │  ゲーム開始時に [2] の世界モデルを履歴に積む
       ▼
   サーバ (aiwolf-nlp-server) と通信しながら対局
```

各段階は完全に独立していて、台本を作り直さない限り [1] は再実行不要、モデルを変えない限り [2] も再実行不要。
**実ゲーム起動時は LLM 呼び出し 0** でキャッシュを読むだけ (= INITIALIZE タイムアウトのリスクなし)。

---

## 2. 各段階の詳細

### [1] 台本生成 (Claude Opus 4.7)

人狼の実プレイログ風 Markdown 台本を **Claude Opus 4.7** で生成。

- **5人村 10 本 / 9人村 10 本** を用意
- 各台本は役職配置・キャラクター設定 (年齢・性別・性格・口調) ・3 日間の対局を含む
- 議論展開・占い CO・投票宣言・最終結果まで完結している
- キャラクターは多様で、論理派・感情派・年長者・気弱型・茶化し型などの混在
- 出力形式は人間が読める Markdown (見出し・表・発話列挙)

これは **manyshot scenarios** と呼ぶ「お手本」の素材。LLM の few-shot / many-shot prompting と同じ発想で、対局の典型例を大量に見せて学習文脈にする。

### [2] 事前準備 (gpt-5.4)

実ゲーム起動を高速化するため、gpt-5.4 で台本分析を **事前に 1 回だけ** 行ってキャッシュ化。

- talk 系 (発話の質を磨く視点) と action 系 (判断材料の視点) で別々に分析させる
- それぞれ「状況 → 議論/判断 の因果対応」「議論展開のパターン」「Day 内 Turn 進行と投票宣言フェーズ」「応答の散らばり方」「主流から外れる位置取り」などの観点で観察を抽出
- 出力された観察まとめ (各 800〜1100 字) をローカルキャッシュに保存

キャッシュは**村サイズ × サーバ仕様 (request-response / グループチャット)** で 4 種類が独立して並走。

### [3] 実ゲーム (gpt-5.2 / gpt-4o-mini)

対局中の役割分担:

| 系統 | 担当リクエスト | モデル | 求められる性質 |
|---|---|---|---|
| **talk 系** | 発話 (TALK) ・ 囁き (WHISPER) | gpt-5.2 | キャラクター演技の自然さ・議論の機微 |
| **action 系** | 投票 (VOTE) ・ 占い (DIVINE) ・ 襲撃 (ATTACK) など | gpt-4o-mini | 対象 1 名を選ぶ判断, 軽量で十分 |

両系統は別の LangChain インスタンスを持ち、会話履歴 (`llm_message_history`) も独立。
キャラクター identity (役職・プロフィール) と日次イベント (前日の処刑・襲撃結果など) は **両系統に同期** して、整合性が取れた状態で動く。

## 3. ゲーム開始前の 3 ターン

実ゲーム起動時、各エージェントは [2] でキャッシュ化した分析を **対話形式** で履歴に積む。
talk 系 / action 系それぞれに同じ構成で積まれる:

```
[0] SystemMessage  (監督 → 役者)
    役割: 出力ルールの伝達
    内容: 「お手本台本を渡す。学ぶのは形式でなく中身。本番では
           talk 系 = 発話本文のみ / action 系 = 対象名のみ出力」

[1] HumanMessage   (監督 → 役者)
    役割: 台本提示 + 分析依頼
    内容: 10 本の実プレイログ全文 + 「状況 → 議論/判断 の因果,
           議論パターン, Day 内 Turn 進行, 応答の散らばり方 を
           観察してまとめて」と観点を指示

[2] AIMessage  (事前生成済みのキャッシュから読み込み)
    役割: 役者の所見まとめ
    内容: 「2-CO で占い理由比較が主題に, 1-CO は結果共有で進む.
           Day 1 最終 Turn は @-mention 付き投票宣言で締める」
           のような 800〜1100 字の観察記述
    ※ talk 系と action 系で観察観点が違うので分析内容も別
```

この 3 ターンが「役者 (LLM) が監督 (システム) から台本を渡され、所見を返した」という会話として履歴に永続化される。
以降のすべての発話・判断は LLM が **自分で書いたこのまとめを参照** して行う。

---

## 4. ゲーム中のフロー

[0]-[2] が履歴に積まれた状態でゲームが始まり、サーバから順次リクエストが届く:

```
INITIALIZE          ← 役職・プロフィール通知
   │
   ├─> DAILY_INITIALIZE  ← Day 開始 (前日の出来事)
   │      │
   │      ├─> TALK x N     ← 議論
   │      ├─> WHISPER x N  ← (人狼間, 9人村以上)
   │      ├─> DIVINE       ← 占い対象選択 (占い師のみ)
   │      ├─> VOTE         ← 投票
   │      └─> ATTACK       ← 襲撃 (人狼のみ)
   │
   ├─> DAILY_FINISH    ← Day 終了 (投票結果)
   │
   ├─> (次の Day へ) ...
   │
   └─> FINISH          ← 終局
```

各リクエストは talk 系 / action 系 のどちらかに振り分けられ (共通リクエストは両方に積む)、
そのときの履歴 + 新規プロンプトで LLM が応答を生成する。

## 5. 設定の主軸

| 設定 | 役割 |
|---|---|
| **mode** | multi-turn (履歴を保持) / single-turn (毎回フルコンテキスト) |
| **lang** | 日本語 / 英語のプロンプト切替 |
| **headings** | プロンプトブロック冒頭の見出し付与 (markdown / xml スタイル) |
| **profile.source** | キャラクター identity をサーバから受け取るかローカル定義から引くか |
| **agent.num** | 村サイズ (5 / 9 / 13 で別キャッシュ) |
| **agent.freeform** | サーバが request/response 仕様か, グループチャット仕様かに応じた挙動切替 |
| **scenario.delivery** | 台本フィードを起動時に一括 (full) するか日別 (by_day) に分けるか |
| **scenario.prewarm** | 事前準備フェーズで使うモデルを実ゲーム用と独立に指定 |
| **llm.separate_langchain** | talk 系 / action 系で LLM インスタンスと履歴を分離するか |
| **prompt.narration_split** | 発話本文を `「」` で囲ませてト書きを許可するモード |

## 6. 何が独自か (manyshot プロジェクトの特徴)

1. **Claude Opus 4.7 で台本そのものを生成** — 多くの few-shot 系手法は人手作成 or 既存ログをそのまま使うが、ここでは LLM 自身に「お手本となる対局」を書かせている (台本品質に対して LLM の創造性を活用)
2. **役割を分けた "監督-役者" 形式の事前学習** — 単なる few-shot ではなく、「役者役の LLM に台本を読ませて自分で観察まとめを書かせる」工程を挟む。出力はそのまま履歴の AIMessage として残るので、本番で LLM は自分の覚書を参照する形になる
3. **キャッシュ機構による起動高速化** — 事前準備の結果を JSON にキャッシュし、実ゲーム起動時はキャッシュ読みのみで LLM 呼び出し 0。サーバの INITIALIZE タイムアウト (60s) を絶対に超えない保証になる
4. **役割別のモデル分離** — talk = gpt-5.2 (演技品質)、action = gpt-4o-mini (軽量判断)、prewarm = gpt-5.4 (要約コスト効率) と、3 段階に異なるモデルを使い分けてコストと品質のバランスを取る
5. **状況条件付きの観察観点** — 単なる「こうしろ」のパターン暗記ではなく「こういう状況ならこう動く」の因果対応で抽出するよう要約プロンプトを設計。状況に紐付かない盲目的な再現を防ぐ
