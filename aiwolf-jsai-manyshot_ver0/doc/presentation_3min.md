## 1. システムのアイデンティティ (1 行で)

**「ゲーム開始前に LLM に “お手本台本” を読ませ、自分で観察まとめを書かせ、その所見を会話履歴に永続化したうえで対局させる」エージェント。**

few-shot prompting の発想を *many-shot* (= 10 本の対局台本) に拡張し、さらに「役者役 LLM の自己要約 → 履歴に AIMessage として残す」という工程を加えたところが独自性。

---

## 2. 3段階パイプライン (これがスライドの骨格)

```
[1] 台本生成 (オフライン)            Claude Opus 4.7
      │  実プレイ風 Markdown 台本を生成
      ▼  data/sample_games_md/ に 5人村×10本 / 9人村×10本
[2] 事前準備 (モデル変更時のみ)       gpt-5.4
      │  台本を読み込み, 自分で分析・要約
      ▼  data/scenario_cache/ にキャッシュ (4種: 5p/5p_freeform/9p/9p_freeform)
[3] 実ゲーム (本番)                  talk: gpt-5.2 / action: gpt-4o-mini
      │  起動時にキャッシュを履歴に積む (LLM 呼び出し0)
      ▼  サーバ (aiwolf-nlp-server) と対局
```

- **段階ごとに違うモデル** を使い分けてコストと品質をバランスさせている。
- **本番起動時は LLM 呼び出しゼロ** → サーバの INITIALIZE タイムアウト (60s) を絶対に超えない設計上の保証。

---

## 3. 「台本を事前に読ませる」とは具体的に何か

実ゲーム開始時、各エージェントの会話履歴 (`llm_message_history`) の先頭に **3 ターン分の対話** を仕込む:

| Idx | 種類 | 中身 | サイズ |
|-----|------|------|--------|
| `[0]` | SystemMessage | 「お手本台本を渡す。学ぶのは形式でなく中身」+ 出力ルール | ~700 字 |
| `[1]` | HumanMessage  | **10 本の対局台本本文 + 分析観点指示** (状況→議論の因果, Turn 進行, 投票宣言フェーズ等) | **~37,000 字** |
| `[2]` | AIMessage     | gpt-5.4 が事前に書いた **自分の観察まとめ** (キャッシュから読み込み) | ~1,500 字 |

ポイント:
- `[1]` の台本本文は **Claude Opus 4.7 が書いた架空対局**。LLM 自身に「お手本」を作らせている。
- `[2]` は監督に渡された役者の所見メモという体裁。以後の発話・判断で LLM は **自分が書いたまとめを参照** する形になる。
- 同じ仕組みを **talk 系 (発話用) と action 系 (投票・占い等) で別観点** で持つ。

---

## 4. なぜこの設計か (発表で強調する 3 点)

1. **few-shot ではなく "監督-役者" 形式** — 単に台本を見せるだけでなく、LLM 自身に「観察まとめ」を書かせて履歴に残す。本番の LLM は自分のメモを参照するので、外から押し付けたルールよりも内面化されやすい。
2. **状況→振る舞いの因果で要約させる** — 「こうしろ」のパターン暗記ではなく「こういう状況ならこう動く」の形で抽出するよう要約プロンプトを設計。盲目的な模倣を防ぐ。
3. **役割ごとに別モデル** — talk 系 (gpt-5.2, 演技品質重視) と action 系 (gpt-4o-mini, 軽量判断で十分) で LangChain インスタンスごと分離。発話の自然さと投票の判断を独立に最適化できる。

## 5. 実装上の要点 (技術スライド用, 必要なら一枚)

- 台本フィードのエントリ点: `src/agent/agent.py::_feed_sample_games()` (`agent.py:760` 付近) — `Agent.initialize()` の冒頭で呼ばれる。
- キャッシュ: `data/scenario_cache/<dir>/<hash>.json` を `src/utils/scenario_cache.py` で読み込む。人間可読版は `data/scenario_cache_readable/`。
- 事前生成スクリプト: `scripts/prewarm_scenario.py` (`config.scenario.prewarm.{talk,action}` で要約モデルを別指定可能)。
- Anthropic 利用時は `apply_cache_control` が起動時 `[1]` の HumanMessage を `cache_control` 対象にし、prompt cache を自動的に効かせる (`src/utils/anthropic_cache.py`)。OpenAI は自動キャッシュで効く。
- talk/action 系の履歴分離は `llm.separate_langchain: true` で有効化。共通リクエスト (`INITIALIZE` / `DAILY_INITIALIZE` / `DAILY_FINISH`) は両系統に同期して整合性を保つ。

## 7. 想定される質問への一行回答

- **Q. なぜ台本まで LLM に書かせる?** — 既存ログだとキャラクター多様性や対局展開の偏りを制御しにくい。Opus 4.7 で生成すると役職構成・性格・口調を均しつつ 10 本揃えられる。
- **Q. 単なる few-shot との違いは?** — few-shot は手本を見せるだけ。本システムは「役者 LLM に自分で観察まとめを書かせ、それを履歴に残す」工程を挟む点が違う。
- **Q. キャッシュは何をしているか?** — 事前準備フェーズの要約結果 (gpt-5.4 出力) を JSON で保存。本番起動時はキャッシュ読みのみで履歴に積めるので、INITIALIZE タイムアウト 60s 内に確実に収まる。
- **Q. talk と action で別モデルにしている理由は?** — talk は演技の自然さ、action は対象 1 名を選ぶだけ。同じモデルだと talk 側の重さが action のコストを押し上げる。役割分離で投票精度はそのままに発話品質を上げる予算配分にしている。
