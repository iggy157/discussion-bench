# INLG構想 — 論文として最も防御的な手法設計（提案）

作成日: 2026-06-21 / 対象: 「LLM生成の完全議論台本＋分析を手本として与え、多人数特有の談話失敗を改善する」研究

> 設計の核となる原則:
> **(P1) 各ドメインは各コミュニティの正準ハーネスに忠実に再現する**（faithfulness-first）。
> **(P2) 介入層（台本／発話例／分析の注入機構・LLM・復号・トークン予算・例数）を両ドメインで完全に一定に保つ**（intervention invariance）。
> **(P3) 失敗様態のトランスクリプト評価指標は両ドメインで同一手順で計算する**（shared measurement）。
> 公平性は「両ドメインのゲーム機構を同一化する」ことではなく、(P2)(P3)で担保する。ドメイン機構の差は2ドメイン設計の前提そのものであり、各正準仕様に従う限り批判されにくい。

---

## 0. なぜ「同一プロトコルに寄せない」のが正しいか（重要な方針転換）

調査で判明した決定的事実:
- **HiddenBenchの正準プロトコルは中央オーケストレータによる逐次ターン制＋構造化された事前/事後回答**であり、自由群チャットではない（4エージェント・T=15ラウンド・round1は順、以降は全員の最新発言を見て応答・固定ラウンド）。出典: Li, Naito & Shirado, arXiv:2505.11556（§4.2）。
- **AIWolfDialの正準プロトコルは同期ターン制**（5人村、1日あたり1エージェント4発話・計20発話、1発話125字（@メンションあれば250字）、1分タイムアウト）。出典: AIWolfDial 2024/2025 概要論文。
- したがって両者を1つのWS自由群チャットに無理に寄せると、**どちらかの正準仕様から乖離**し「本当にHiddenBench/人狼をやったのか」と査読で突かれる。

→ 結論: **transport（WebSocket）と"頭脳"（エージェントのプロンプト構築＋注入＋LLM呼び出し＋コスト計上）は共有**するが、**進行ロジック（ターン順・回答収集・終了条件）は各ドメインの正準仕様に忠実な別アダプタ**にする。これが faithfulness と再現性を両立する唯一の設計。

### 2.2 サーバ
- 新規 軽量Python **bench-server**（develop配下に新設、最終的にdemoへ移植）。`benchmark.json` を直接読み、上記プロトコルを忠実実装。

- 事前/事後回答は新規リクエスト型（`PRE_ANSWER` / `POST_ANSWER`）として最小追加（エージェント側調査で「~2リクエスト型・数十行」と確認済み）。

### 2.3 タスク成功指標（HiddenBenchネイティブ）
- 事後精度（主）・事前精度・Full Profile精度・統合ゲイン・集団推論ギャップ（§2.1）。

## 3. 介入の操作化（3×2の6条件）— ここが本研究の心臓

| | 分析なし | 分析あり |
|---|---|---|
| ベースライン | ① ベースライン | ② 分析のみ |
| 発話few-shot | ③ 発話few-shot | ④ 発話few-shot＋分析 |
| 台本few-shot | ⑤ 台本few-shot | ⑥ 台本few-shot＋分析（本命） |

### 3.1 各手本の定義（両ドメイン共通の作り方）
- **台本（script）**: 始め〜終わりの**完全な多人数議論トランスクリプト**。
  - 人狼: 5人村の1ゲーム全文（manyshot_ver0の `data/sample_games_md/` 形式）。
  - HiddenBench: 1タスクの discussion 全文（4エージェント×15ラウンド＋事前/事後）。
- **発話few-shot**: 同じ台本群から切り出した**単一発話単位**の例（議論全体の流れを含まない）。
- **分析（analysis）**: 台本から抽出した「良い議論はこう進む／ここを見るべき」という**要点記述**（命令ではなく観察）。

### 3.2 漏洩・話題依存の排除（査読最重要）
- **(L1) held-out**: 手本に使う台本/発話例/分析は、**評価インスタンスとは別の**ゲーム/タスクから生成する。HiddenBenchは65タスク中、評価に20タスク使うなら手本は**残り45タスク側**から作る（タスクID重複ゼロを機械的に保証）。人狼も評価ゲームと別シードのゲームから作る。
- **(L2) 話題非依存**: 分析は「議論の進め方・着目点」という**話題非依存の記述**に限定し、特定タスクの正解選択肢や固有事実を含めない（含めると正解漏洩）。レビューで分析テキストを検査し正解語・タスク固有名を除去。
- **(L3) 生成器固定・系列分離**: 台本・分析の生成は **Claude固定**。議論エージェントは **Gemma**（別系列）→ 自己模倣交絡を排除（INLG構想の主張どおり）。審査LLMは **GPT**（さらに別系列）→ LLM-judgeの自己選好バイアスを最小化。3系列を分離。

### 3.3 トークン量の統制（純粋比較の担保）
- **(T1) 対応付け**: ③発話few-shot ↔ ⑤台本few-shot を**入力トークン数で一致**させる。④発話few-shot＋分析 ↔ ⑥台本few-shot＋分析 も一致させる。
- **(T2) 計測**: 一致は**議論エージェント（Gemma）の実トークナイザ**で測る（文字数でなくトークン数）。各セル×各ドメインの実トークン量を**論文に表で明示**。
- **(T3) 自然な少トークン**: ①ベースライン・②分析のみは自然に少トークンになる（効果が情報量でなく要点の質である傍証）。これも表に明示。
- **(T4) 同一注入機構**: 6条件すべて、manyshot_ver0の `_feed_scenario_chunk()` 同一経路で注入（HumanMessage=本体、AIMessage=分析 or 静的ack）。条件間で注入機構・履歴位置・キャッシュ方式を変えない。

### 3.4 PRICoTとの差別化（敬意ある引用）
PRICoT（Yamazaki et al., INLG2025, `2025.inlg-main.35`）も「成功/失敗の事後分析を抽出し後段に注入」する点で近縁。**本研究の差分を4軸で明記**:
1. PRICoTは**正解ラベル必須**（教師あり成功/失敗分類）／本研究の分析は正解ラベル非依存。
2. PRICoTは**再利用可能な汎化原則を蒸留**して別問題へ転移／本研究は「良い議論の進め方」という手本提示。
3. PRICoTは**ベクトルDBで事例ごとに検索注入**／本研究はゲーム開始時に一括提示（検索バンク無し）。
4. PRICoTは**二段(オフライン構築→オンライン検索)**パイプライン／本研究は単段の文脈内手本。
→ 「②分析のみ／④⑥＋分析」はPRICoTの近縁だがこの4点で異なる、と本文で宣言。

## 4. 評価方法（両ドメイン同一手順 = 公平性の背骨）

### 4.1 主観評価（AIWolfDial準拠ルーブリックを両ドメインに適用）
AIWolfDialの A–F は多年安定の community standard。**両ドメインのトランスクリプトに同一ルーブリックを適用**:
- A 発話の自然さ / B 文脈の自然さ / C 一貫性・無矛盾 / D 行動と発話の整合 / E 表現の多様性・キャラクタ一貫性 /（多人数村のみ）F チームプレイ。
- **尺度の選択（未決定B）**: 2024=絶対5段階Likert / 2025=順位付け。**絶対スコアで条件間比較したいなら2024式Likertを採用**（6条件の絶対比較に向く）。順位法は同点禁止で条件比較に不向き。→ 推奨: **5段階Likert**＋評価者間一致(κ/α)を報告。
- **LLM-as-judge**: AIWolfDial2025がGPT-4o/GPT-5で同ルーブリックを再現し人手と高相関（A,B,D,E）を確認済み。本研究も**GPTを審査LLM**にして人手主観評価の補助＆スケール手段とする（系列分離=§3.2 L3）。
- HiddenBenchのDは「行動＝最終回答と発話の整合」に読み替え（ドメイン適応を明記）。

### 4.2 客観評価（失敗様態ごと・正確な出典と自作明示）
> 調査で判明: いくつかの「指標」は元論文に式が無い。**自作式は自作と明記し、適切な原典を引く**ことが防御の要。

| 失敗様態 | 採用指標と定義 | 出典の扱い |
|---|---|---|
| 情報の出し惜しみ | **情報表面化率** = (台本/タスクの未共有事実のうちトランスクリプトで1回以上言及された種類数)/(未共有事実総数)。検出はルール照合＋LLM抽出。**＋結果指標として HiddenBench事後精度**。 | **HiddenBenchは表面化率の式を定義していない** → 自作と明記。リコール/言及測度の原典として Stasser & Titus 1985 (JPSP, DOI:10.1037/0022-3514.48.6.1467)、coverage式は Lu, Yuan & McLeod 2012 を引用。精度系のみ HiddenBench(Li et al.) を引用。 |
| 早期収束 | **収束ラウンド** = 全エージェントが同一選択肢に一致した最小ラウンド。**重要事実表面化前合意率** = (合意成立ラウンド < 全重要未共有事実が出揃うラウンド) の割合。 | **HiddenBenchは収束ラウンド式を持たない**（記述: 「最初の2ラウンド/8発話で誤合意 vs 人間平均53.4発話」は記述統計）。最終ラウンド合意率は Smit et al. ICML2024 (arXiv:2311.17371)、ラウンド分解は Wu et al. (arXiv:2511.07784) を引用。閾値式は自作と明記。 |
| 思考停滞・多様性 | **distinct-1/2**（Li et al. NAACL2016, arXiv:1510.03055; 分母=総トークン数か総n-gram数かを実装で明記）。**自己反復多様性** = 100−Self-BLEU(初期案,改訂案)（Liang et al. EMNLP2024, arXiv:2305.19118）。立場多様性は別途。 | **DMADは意味的多様性/Self-BLEU/distinct-nを定義していない**（DMADの多様性はstrategy coverageで正解必須・トランスクリプトから不可）→ DMADを意味的多様性の出典に引かない。distinct-nはLi et al.、Self-BLEU系はLiang et al.。 |
| 反射的同調 | **同調率/独立率** をBenchForm式で。各ラウンド全エージェントの立場をポーリング(LLM抽出)し、(他者多数と異なる正解→多数に合わせ誤りへ反転)の割合を計算。 | BenchForm (Weng et al. ICLR2025, arXiv:2501.13381) のCR/IRを引用。**IR≠1−CR**（連言的頑健性）に注意。ラウンド遷移の taxonomy は Talk Isn't Always Cheap (arXiv:2509.05396)。BenchForm原設定はscripted confederate＋推論QAなので、自由議論への適応を明記。 |

- **両ドメイン適用の正直な線引き**:
  - distinct-n/Self-BLEU多様性・同調率・収束ラウンドは**自然にドメイン汎用** → 両ドメインで同一計算（P3）。
  - 情報表面化はHiddenBenchネイティブ。**人狼の類似構成（私的情報の開示: 占いCO・占い結果・役職主張の表面化率）**を「ドメイン適応版」として報告し、HiddenBench版と同列に主張しない（正直な framing）。

## 5. モデル・規模・要因計画

- **議論エージェント**: Gemma の上位/下位2サイズ（規模頑健性を安価・再現性高く検証。生成器Claudeと別系列で自己模倣交絡なし）。**未決定C**: 具体サイズ（例 Gemma-2-27b / 9b など）。
- **生成器**: Claude固定（台本・分析）。**審査LLM**: GPT固定。
- **規模**: 6条件 × 2ドメイン × 2モデルサイズ × 20ゲーム = **240ゲーム**。
  - HiddenBench: 20タスク＝65中の20の**異なる**タスク（偏り回避）。手本は残り45タスク側から（§3.2 L1）。
  - 人狼: 20ゲーム＝別シード。
- **復号統制**: 温度・max_tokens等を全条件・全ドメインで固定し明記（HiddenBench論文は温度を明示せず、repo既定は0.7 → 本研究で固定値を宣言）。
- **協調〜対立5条件**（HiddenBench）は主実験では1つ（中立的な Cooperative 相当）に固定し交絡を避ける。5条件掃引は付録のロバストネスに回す（任意）。

## 6. システム/ディレクトリ構成（manyshot_ver0軸、最終demo移植）

```
develop/
  agent/      # 議論エージェント（共有頭脳）。HiddenBenchアダプタを最小追加
    src/agent/agent.py            #   + pre_answer()/post_answer()/discussion-turn handler
    prompts/<lang>/hiddenbench/   #   + HiddenBench用テンプレ・分析テンプレ
    data/exemplars/               #   + 台本/発話例/分析（held-out, 6条件分）
    config/condition_*.yml        #   + 6条件 × 2ドメインのconfig
  aiwolf-nlp-server/              # 人狼サーバ（無改造）
  server/hidden-bench/ (新設)      # HiddenBench忠実進行サーバ（benchmark.json直読）
  inlg-compose/ (新設)            # docker compose: yaml1つでaiwolf/hiddenbench選択＋同時起動
                                  #   （aiwolf-nlp-demoのservice名・複数サーバ並走パターンに準拠）
```
- compose は aiwolf-nlp-demo の「同一transportのサーバを service名で複数同時起動」パターンに乗せる。
- 介入層（`_feed_scenario_chunk()`・LLM呼び出し・コスト計上）は両ドメインで**同一コード**を通す（P2の実装上の保証）。
