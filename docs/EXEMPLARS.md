# Exemplars — where they go and how to build them

`agent/<env>/exemplars/` is where the **examples** used by the six conditions are placed:
three kinds (scripts, utterance-fewshot, analysis), organized by environment (werewolf /
HiddenBench) and language (en / jp). They are intentionally **empty** for now — drop files in
later. A condition with no examples behaves exactly like `baseline`. The full condition design
is in [METHODOLOGY.md](METHODOLOGY.md).

## Where each kind goes

```
agent/<env>/exemplars/<lang>/<kind>/
  env  : aiwolf | hidden-bench
  lang : en | jp
  kind : scripts     -- full discussion transcripts (script few-shot; conditions ⑤⑥)
         utterances  -- single-utterance examples (utterance few-shot; conditions ③④)
         analysis    -- "where to look / how a good discussion proceeds" notes (conditions ②④⑥)
```

Which condition reads which kind:

| # | condition key | scripts | utterances | analysis |
|---|------------------------------|:-------:|:----------:|:--------:|
| ① | `baseline`                   |    –    |     –      |    –     |
| ② | `analysis_only`              |    –    |     –      |    ✓     |
| ③ | `utterance_fewshot`          |    –    |     ✓      |    –     |
| ④ | `utterance_fewshot_analysis` |    –    |     ✓      |    ✓     |
| ⑤ | `script_fewshot`             |    ✓    |     –      |    –     |
| ⑥ | `script_fewshot_analysis`    |    ✓    |     –      |    ✓     |

## Controls you must keep

Examples decide the validity of the study, so keep these four when authoring them:

1. **Build from a disjoint set (leakage control).** Examples must come from games/tasks
   different from the evaluation set. For HiddenBench, if you evaluate task X, build scripts
   only from the other tasks.
2. **Topic-independence.** `analysis/` must describe *how a good discussion proceeds / where
   to look*, not the correct option or task-specific facts (that would leak the answer).
3. **Token-matching.** `utterances/` (③) and `scripts/` (⑤) are token-matched, and the
   `+analysis` pair (④, ⑥) too — measured with the **discussion agent's** tokenizer. Record
   the per-condition token counts.
4. **Cross-family generation.** Scripts/analysis are generated by **Claude**, the discussion
   agent is **Gemma**, the judge is **GPT** — three separate families.

## File format and loading

- `scripts/*.md` — one full transcript per file (same Markdown convention as
  `agent/data/sample_games_md/`).
- `utterances/*.md` (or `.txt`) — short single-utterance examples, one per block.
- `analysis/*.md` — the analysis text attached to the example.

These are read through the **same injection mechanism** (`_feed_scenario_chunk()`) for both
werewolf and HiddenBench. Using one code path for both is what guarantees cross-domain fairness
at the implementation level (methodology P2). The mapping from condition to example folder is
defined in `config/conditions.yml` (and the Japanese reference copy `config/conditions.ja.yml`).
