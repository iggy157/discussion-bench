# 手本（exemplars）の置き場と作り方

`agent/<環境>/exemplars/` 以下は、6条件の実験で使う**手本**を置く場所です。台本・発話例・分析の3種類を、
環境（人狼／HiddenBench）と言語（en／jp）ごとに収めます。現状は意図的に**空**にしてあり、あとからファイルを
置く前提です。手本を1つも置いていない条件は、自動的に `baseline` と同じ挙動になります。条件の全体設計は
[INLG_METHODOLOGY.md](INLG_METHODOLOGY.md) を参照してください。

## どこに何を置くか

```
agent/<環境>/exemplars/<言語>/<種類>/
  環境 : aiwolf | hidden-bench
  言語 : en | jp
  種類 : scripts     -- 完全な議論トランスクリプト（台本few-shot。条件 ⑤⑥）
         utterances  -- 単一発話の例（発話few-shot。条件 ③④）
         analysis    -- 「どこを見るべきか／良い議論はどう進むか」の分析（条件 ②④⑥）
```

どの条件がどの種類を読むかは、次の対応になっています。

| # | 条件キー | scripts | utterances | analysis |
|---|------------------------------|:-------:|:----------:|:--------:|
| ① | `baseline`                   |    –    |     –      |    –     |
| ② | `analysis_only`              |    –    |     –      |    ✓     |
| ③ | `utterance_fewshot`          |    –    |     ✓      |    –     |
| ④ | `utterance_fewshot_analysis` |    –    |     ✓      |    ✓     |
| ⑤ | `script_fewshot`             |    ✓    |     –      |    –     |
| ⑥ | `script_fewshot_analysis`    |    ✓    |     –      |    ✓     |

## 守るべき統制

手本は実験の妥当性を左右するので、作るときに次の4点を必ず守ります。

1. **評価と別インスタンスから作る（漏洩防止）。** 手本は、評価に使うゲーム/タスクとは別のものから作ります。
   HiddenBenchで例えばタスクXを評価に使うなら、台本はそれ以外のタスクからのみ作ります。
2. **話題に依存させない。** `analysis/` は「議論の進め方・着目点」に限定し、正解の選択肢やタスク固有の事実は
   書きません（書くと答えが漏れます）。
3. **トークン量を揃える。** `utterances/`（③）と `scripts/`（⑤）はトークン数を一致させ、`+analysis` の組
   （④と⑥）も一致させます。測るのは**議論エージェントのトークナイザ**で、各条件のトークン数を記録しておきます。
4. **生成・議論・審査を別系列のモデルに分ける。** 台本・分析はClaudeで生成し、議論エージェントはGemma、審査役は
   GPT——という3系列の分離を保ちます。

## ファイル形式と読み込まれ方

- `scripts/*.md` — 1ファイルに1つの完全トランスクリプト（`agent/data/sample_games_md/` と同じMarkdown形式）。
- `utterances/*.md`（または `.txt`）— 短い単一発話の例を1ブロックずつ。
- `analysis/*.md` — 手本に添える分析テキスト。

これらは人狼でもHiddenBenchでも**同じ注入機構**（`_feed_scenario_chunk()`）を通って読み込まれます。両環境で
同一経路を通すこと自体が、ドメイン間の公平性（方法論の P2）を実装面で保証しています。条件と手本フォルダの
対応づけは `config/conditions.yml`（および日本語版 `config/conditions.ja.yml`）で定義されています。
